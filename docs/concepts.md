# 概念与设计

> [上手教程](getting-started.md) · [操作指南](how-to.md) · [参考手册](reference.md) · [概念与设计](concepts.md) · **记忆库使用**

解释这个工具为什么长这样。不教操作（那是[操作指南](how-to.md)的事），只讲设计与取舍。

## 要解决的问题

面向规模较大的多文档语料（设计目标为百亿字级），核心诉求是两个：

1. **找得准**：在海量原文中定位到具体几句，而不是「这篇文档里大概有」。
2. **信得过**：结果能回溯到原文出处，不被模型臆测替代。

后面所有设计都从这两条推出来。

## 条件树与稳定地址

把原文整理成树，每个节点有形如 `1-3-2` 的地址。地址不是存死的编号，而是由 `parent_id + sort_order` 动态重算——增删、拖拽、重挂之后，地址依然按当前结构成立。于是每个句子都有一个**可引用的坐标**：检索结果、agent 回答、节点间引用、历史 diff 都用地址说话，人能照着地址核对。

节点级粒度是刻意的：导入时就切到句子级，粒度在导入期锁定。检索命中是「第几个节点」而不是「第几页」，代价是导入前要把结构定好——这笔交易换来的是引用永远精确。

## 双密度阅读

同一份内容在两种密度间切换：折叠时是 Markdown 文档，展开时是可操作的条件树。两种视图共享同一份原文——导入时保留句子到原文的 offset 映射，存的是映射不是拷贝。原文本身（`source_documents` / `source_spans`）只记 offset，不重组正文结构，所以任何视图都能溯回到源文的字节位置。

## 证据纪律

内置 agent 回答事实问题时，必须先读取正文证据并给出证据节点地址，不以模型常识或题目措辞作答。检索命中只用于挑候选；回答前用 `read` 取回正文。这条纪律同样写进了 MCP 工具的 description，期望外部 agent 遵守同一动线：`find` 挑候选 → `read` 取证据 → 回答附地址。

## 信任分级

节点分两级：**受控**与**不受控**。受控 = 经人工审批的结论；不受控 = 机器产物未经人审。这是「是否过了人这一关」的时态属性，不是质量评分——智能导入、agent 写入、事件记忆卷的产物一律不受控，不存在「写得好就升级」的通道，升级的唯一方式是人审。

ERROR 是一类特殊节点：命中未解决的 ERROR 时，agent 必须停下向用户报告，不得绕过续跑。

## 编辑模型：分支与三方合并

agent 对主库没有直接写权限。结构性修改先写进**编辑分支**（owner=llm 的影子分支），人在应用内逐条或整批裁决后才进主库。

保存（合入主干）按三方模型处理，对账单位是**稳定节点 id**而不是地址（地址会随结构变化漂移）。树的比对用 merkle 哈希加速：子树哈希相同就整棵跳过。三种结局：

- **快进**：主干在分支期间没动，直接写回。
- **受阻（blocked）**：主干发生了结构性失配——删了被改的节点、并发移动、内容漂移。整体拒绝，提示「主干已被修改，无法保存」，只能放弃该分支。不做自动结构调和。
- **冲突（conflicts）**：字段级冲突，弹出冲突对话框由人逐条裁决。agent 不自动解冲突——合并是只有人能拍板的动作。

非快进时的验证是乐观并发：逐条账目点查，代价与改动数成正比，与文档大小无关。冲突与受阻时主干与分支都不动。

## 文档编辑模式

每个文档处于三种模式之一：**只读**、**增量编辑**、**完整编辑**。

- 完整编辑走上面的分支模型，适合人或 agent 修订既有内容。
- 增量编辑只允许**流式追加**（`push`）：消息节点直接进库不走分支，关键词与语义索引随写增量维护。适合聊天记录、日志这类只增不改的数据流。
- 两者互斥：流式写入期间不能分支编辑。要修订流式文档，先切到完整编辑，改完再切回来。互斥不是实现偷懒——追加流与三方合并的并发语义不能同时成立。

海量导入有专门的 `bulk` 会话：临时切异步写、结束恢复安全设置。它在共享后端上要求独占，崩溃窗口由地址校验加幂等重推兜底，属于「专门导入阶段」的工具而非常态。

## 记忆三层

库按时态分三层，结构同构（同一套树地址与检索动词）、语义异质：

| 层 | 回答的问题 | 性质 |
| --- | --- | --- |
| 完整记忆（事件卷） | 确曾发生过什么 | 一个 session 一卷；只追加，封卷后不可变，永不删除 |
| 长期核心记忆 | 现在如此 | 由事件卷提炼、经人工审批的当前事实，每条回指事件出处 |
| 知识文档 | 未来可用 | 导入的资料文档 |

事件卷是**原料**不是结论：agent 投递的是自述日志（用户原话逐字、任务结果、失败教训、可复用结论），提炼成当前事实是另一条链的事，且必须过人审——这与信任分级是同一条原则在记忆侧的投影。卷按 24 小时节律自动封卷、进入可提炼。

时间纪律：召回结果附时间元数据，同主题证据冲突时新者胜，不得以「内容看起来更合理」推翻时间新旧——合理感是模型先验，时间是客观信号。

当前版本（0.2.0）落地的是第一段链路：投递、封卷、列卷可用；提炼与审批链路在后续版本。

## 共享后端

SQLite 是单写者数据库，而这里至少有三类客户端：应用、MCP server、命令行脚本。解法是**一库一后端**：数据库由一个 headless 后端进程独占，所有客户端连它。先到的客户端拉起后端；后来者按数据库绝对路径哈希派生的命名管道接入（描述文件放数据库同目录，丢失也能在同一管道名会合）。于是应用开着的同时，外部 agent 可以通过 MCP 操作同一个库，互不踩踏。

## 本地优先

文档、结构、向量、历史、记忆全部在本地（SQLite + LanceDB），不依赖云服务。需要网络的只有两件事且都是可选项：调 LLM 接口（摘要 / agent / 智能导入），以及下载向量模型。MCP 是数据的唯一对外边界，权限档在启动时锁定。
